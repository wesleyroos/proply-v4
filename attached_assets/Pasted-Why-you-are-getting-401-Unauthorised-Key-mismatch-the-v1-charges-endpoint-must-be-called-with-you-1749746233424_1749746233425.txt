Why you are getting 401 Unauthorised
Key mismatch – the /v1/charges endpoint must be called with your secret key (sk_test_…) in the Authorization header. If the key was revoked or copied with an extra space it is rejected.
Wrong request body – /v1/charges expects a token (e.g. tok_abc123) in the source field, not raw card data. Sending card details directly is blocked for PCI‑DSS reasons and will always fail.
Old vs new endpoints – Yoco’s newer docs focus on the hosted Checkout flow (https://payments.yoco.com/api/…). The older direct‑charge endpoints (https://api.yoco.com/v1/…) still work, but only when used exactly as documented (token + secret key).
“Prepend Bearer to the secret key and insert it into the Authorization header.”
— Yoco Developers – Authentication 
(developer.yoco.com/online/api-reference/authentication)
Correct flow to store a card and charge it later
Step	Where it runs	Key to use	Yoco call	What it returns
1. Collect card details & create a token	Front‑end (browser / mobile app)	Publishable key (pk_test_…)	POST https://api.yoco.com/v1/tokens (or easier: Yoco Web SDK yoco.inline.createToken())	{ id: "tok_xxx", … }
2. Store tok_xxx in your DB	Your server	–	–	–
3. Charge the saved token	Back‑end	Secret key (sk_test_…)	POST https://api.yoco.com/v1/charges with body { amount, currency:"ZAR", source:"tok_xxx" }	Payment object
Sample code snippets
1. Front‑end (tokenisation)
html
<script src="https://js.yoco.com/sdk/v1/yoco-sdk-web.js"></script>
<script>
  const yoco = new window.YocoSDK({ publicKey: 'pk_test_12345' });

  async function pay() {
    const { error, token } = await yoco.inline.createToken({
      amountInCents: 100,
      currency: 'ZAR',
    });
    if (error) {
      console.error(error.message);
      return;
    }
    // Send token.id to your server
    fetch('/charge', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ token: token.id }),
    });
  }
</script>
2. Back‑end (charge the token)
bash
curl https://api.yoco.com/v1/charges \
  -H "Authorization: Bearer sk_test_37cc036dy8nVo4R0eb4403ea06e0" \
  -H "Content-Type: application/json" \
  -d '{
        "amount": 100,
        "currency": "ZAR",
        "source": "tok_test_visa_4242",
        "description": "Order #1234"
      }'
Answers to your specific questions
Question	Answer
Is the test secret key format correct?	Yes – it should start with sk_test_. Make sure there are no spaces and the key is active.
Are we using the right endpoint?	/v1/charges is correct only for charging an existing token. For tokenisation use /v1/tokens (or the Web SDK).
Extra headers needed?	No. Only Authorization: Bearer <secret_key> and Content-Type: application/json.
Separate tokenisation endpoint?	Yes – /v1/tokens (client side) or the Web SDK which wraps it.
Does the test key need activation?	Test keys work immediately. If in doubt, regenerate them in the Business Portal and update both publishable & secret keys.
Why do we still get 401?	Most common causes: key copied incorrectly, using publishable key instead of secret key on the server, or including raw card data instead of a token.
Next steps for your web‑app
Regenerate a fresh pair of test keys in the Yoco Business Portal.
Use the Web SDK (or direct /v1/tokens call) on the client to obtain tok_….
Call /v1/charges on your server with the token and your secret key.
When you are ready for production, switch to the live keys (pk_live_…, sk_live_…).
